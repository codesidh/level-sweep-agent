§1

Executive Summary

LevelSweepAgent is a fully automated execution platform that trades 0DTE SPY options under the four-level liquidity-sweep strategy (PDH / PDL / PMH / PML) gated by an EMA(13/48/200) and ATR(14) confluence filter. The platform is presented to the user as "an AI agent that trades for them"; internally, it is a deterministic execution core with a Claude-powered AI overlay that supervises, narrates, and converses — but cannot place orders.

Why hybrid. A pure-LLM trader would be non-deterministic, slow (1–3 s per call), prone to hallucinated orders, expensive, hard to audit, and indistinguishable from "investment advice" for regulators. A pure-deterministic trader has none of those problems but cannot reason about narrative-level signals (news, regime shifts, anomalous fills). The hybrid model — deterministic core plus AI overlay — keeps the order path replayable bar-for-bar and sub-250 ms, while the AI adds explanation, supervision, and conversation. (See ADR-0002.)

Phasing. Phase A is single-tenant (owner-only) operation. Phase B (multi-tenant SaaS, paying customers, public UI) is blocked on legal counsel completing RIA / broker-dealer review and a 60-session live soak. Phase B code paths exist behind feature flags, default OFF. The architecture is multi-tenant-shaped from line one: every persistent entity carries tenant_id, every Kafka topic is keyed by tenant, every secret is scoped per tenant.

Operating discipline. Five hard guardrails: (1) AI cannot place orders (2) Fail-closed on the order path (3) Replay parity ≥ 99 % on 30 historical sessions (4) Per-phase production-readiness gate — 5 RTH soak sessions on live deps before next phase merges (5) No real money in tests; financial tables WORM-eligible from day 1.

Cloud & cost. Microsoft Azure (eastus primary, eastus2 cold-standby DR), AKS for containerisation, Strimzi-managed Kafka, MS SQL for system-of-record, MongoDB for audit, Anthropic Claude API for AI. Phase A run-rate ~ $1.4 K / month infra plus ~$99/mo Alpaca Algo Trader Plus and ~$5/day capped Anthropic spend per tenant.

Headline risks. (a) Alpaca outage at 09:30 ET cascade — mitigated by fail-closed Risk FSM and CB cascade; (b) Anthropic 529 burst at market open — mitigated by Sentinel fail-open default and prompt cache; (c) regulatory drift to "investment advice" framing — mitigated by tool-surface design that only explains a user-configured strategy. Full register at §22.

Decision asked of EA / CIO. Approve the Phase A architecture as the build target. Phase B re-review will be requested after legal sign-off and the 60-session live soak.

Business Context & Strategic Drivers

2.1 Mission

Build an AI-agentic execution platform for a single, codified trading strategy that operates without human intervention during market hours and that can — once legal and operational gates clear — be productised as a multi-tenant SaaS. The strategy is the four-level liquidity-sweep on 0DTE SPY options, validated by an EMA / ATR trend-stack. The agent's value is discipline at machine speed: it never deviates, never sizes up after a loss, never trades through news, and never holds past 15:55 ET.

2.4 Constraints

  • Regulatory. Phase A is private use only. Phase B requires RIA / broker-dealer posture review by external counsel; until that resolves, Phase B is feature-flagged off.

  • Schedule. Per-phase soak gates (5 RTH sessions on live deps) cannot be shortcut. P0 / P1 incidents during soak reset the counter.

Architecture Principles

Every design choice in this document traces back to one of the ten principles below. They are listed in priority order — when they conflict, the lower-numbered principle wins.

#PrincipleStatement & Rationale
1Fail-closed on the order pathAny uncertainty — degraded dep, malformed event, missed heartbeat — halts new entries. Existing positions continue under deterministic stop-loss / trailing rules. Why: in trading, doing nothing is almost always cheaper than doing the wrong thing.
2AI is advisory, never authoritative on order placementOnly the Execution Service places orders. The AI's only write into the trade saga is a confidence-gated VETO (≥ 0.85). Why: regulatory framing, hallucination risk, latency.
3Determinism is the contractSame inputs ⇒ same outputs. Replayable from journal. Decision Engine changes require ≥ 99 % parity on 30 historical sessions. Why: auditability, debuggability, reproducible incident analysis.
4Idempotency end-to-endEvery external command carries a deterministic client-id; retries are safe by construction. Alpaca client_order_id = sha256(tenant_id|trade_id|action). Why: at-least-once delivery is the only honest contract Kafka can give us.
5State machines as codeEvery lifecycle (Session, Trade, Risk, Order, Position, Connection) is an explicit FSM with persisted transitions doubling as audit trail. Why: implicit state is invisible; explicit state machines make incident review tractable.
6CAP per domain, not per systemOrder placement / risk budget = strict CP. Audit, dashboard, projection = AP. Why: a one-size CAP profile either over-pays (CP for audit reads) or breaks safety (AP for budget).
7Multi-tenant from line oneEvery persistent entity carries tenant_id; every Kafka topic is keyed by tenant; every secret is scoped per tenant. Why: retrofit is painful and almost always introduces leakage; doing it from day one is free.
8Hot-path latency over service purityIndicator + Signal + Risk + Strike + Saga are in-process within the Decision Engine — not microservices — because four Kafka hops on the inner decision burns the latency budget. Why: we serve a 250 ms SLO; we don't serve REST evangelism.
9Observability is a first-class featureEvery decision and non-decision is logged with full input context, propagated trace_id, and ET local-time stamps. Why: if it isn't observable, it didn't happen — for both incident response and regulatory inquiry.
10Time disciplineAll scheduling and FSM time gates use ZoneId.of("America/New_York") (not fixed-offset EST). Persisted as UTC Instant plus ET local time. JVM tzdata kept current. Replay corpus includes a spring-forward and a fall-back session. Why: DST and fixed-offset bugs are the most common silent killers in financial systems.
Trade-off note Principle 8 (in-process hot-path composition) is the single biggest deviation from textbook microservices. It is consciously chosen and bounded: the in-process Decision Engine is one service that hosts five logical components. We pay this price to hit the 250 ms order-path SLO that the strategy requires.

Trading Strategy & Decision Engine

This section codifies the strategy at engineering precision. It is the canonical reference that the Decision Engine implements; replay parity ≥ 99 % is measured against this specification.

9.1 Universe & Vehicle

ItemPhase A
InstrumentSPY only
Vehicle0DTE options (CALL or PUT)
Strike ruleFirst in-the-money strike
ConcurrencyOne open position at a time (no pyramiding)
Liquidity gateReject strike if bid-ask spread > $0.10 absolute or 5 % of mid

9.2 Reference Levels (finalised by 09:29:30 ET)

LevelDefinitionWindow
PDHPrevious-day HighRTH only · 09:30–16:00 ET prior day
PDLPrevious-day LowRTH only · 09:30–16:00 ET prior day
PMHPre-market / overnight High16:01 ET prior day → 09:29 ET today
PMLPre-market / overnight Low16:01 ET prior day → 09:29 ET today

9.3 Trend Filter (2-min EMA stack)

BULLISH stack (CALLs allowed):
  EMA13 > EMA48 > EMA200
  AND  gap_top    = EMA13 − EMA48  >= 0.3 × ATR(14)
  AND  gap_bottom = EMA48 − EMA200 >= 0.3 × ATR(14)

BEARISH stack (PUTs allowed):
  EMA13 < EMA48 < EMA200
  AND  |gap_top|    >= 0.3 × ATR(14)
  AND  |gap_bottom| >= 0.3 × ATR(14)

Otherwise → choppy regime → NO TRADE

EMA48 hold-exception (used in stop-loss). If |EMA13 − EMA48| < 0.5 × ATR(14), the stop reference shifts from EMA13 to EMA48.

9.4 Sweep Validation

BUFFER = 0.2 × ATR(14)

A level is "swept" only when a 15-min candle CLOSES beyond it by ≥ BUFFER.
Bare wicks or tick-crosses do NOT qualify.

9.5 Entry Decision Matrix (evaluated at each 15-min close from 09:45 ET)

9:45 candle closeEMA stackAction
≥ PMH + BUFFERBullishEnter CALL → target PDH
PML ≤ close ≤ PMH (mid-range, bouncing off PML)BullishEnter CALL → target PMH
≤ PML − BUFFERBearishWait for 10:00 ET candle confirmation
Inside levels w/o boundary breakanyNO TRADE; re-evaluate next 15-min close

Below-PML confirmation rule. If 9:45 closes ≤ PML − BUFFER: at 10:00 ET, if 15m_close ≤ PDH → Enter PUT → target PDL; else NO TRADE.

Continuation scans. If no trade fires at 9:45 / 10:00, the engine continues evaluating each subsequent 15-min close (10:15, 10:30, … cutoff) using the same matrix on the latest level state. Entry cutoff: no new entries after 14:30 ET.

9.7 Risk Management

  • Per-trade size — 5 % × start-of-day portfolio equity (computed once at 09:29 ET; fixed for the day).
  • Daily risk budget — 2 % × start-of-day equity.
  • Re-entry rule — allowed throughout the day while cumulative_realized_loss_today < daily_risk_budget. On breach, HALT new entries; open positions continue under §9.6.
  • Halt persistence — terminal for the day; reset at next session's 09:00 ET.

11.2 Latency-Driving Choices

  • In-process Decision Engine — avoids 4 Kafka hops on the inner decision.
  • GraalVM native image for hot-path services — eliminates JIT warm-up cost on first trade.
  • ZGC for sub-10 ms GC pauses on hot-path JVM (when not native).
  • Disruptor for in-process queue between Indicator → Signal → Risk → Strike → Saga.
  • Kafka acks=all + idempotent producer on critical topics.
  • Pinned consumer threads per partition to avoid scheduler jitter.
  • Sentinel uses Claude Haiku for sub-second response on the binary veto decision.
  • Anthropic prompt caching for repeated context (system prompt + tool defs).

13.3 Consumer-Group Strategy

  • One consumer group per logical service (decision-engine, execution-service, journal-service, ai-agent-service).
  • Consumer instances within a group divide partitions; partition stickiness preserves single-writer-per-tenant guarantees.
  • Phase A: 1 instance per service is sufficient.
  • Phase B: scale instances up to partition count.

13.4 Thundering-Herd Defences

  • Per-(tenant, symbol) partitioning — one tenant's burst can't starve others.
  • Per-tenant rate limit at APIM (e.g., 100 req/min/tenant on dashboard).
  • Bulkhead pools in each service per upstream tenant.
  • Request collapsing on dashboard reads (5 s cache).
  • Backpressure at consumer: pause partition assignment when local queue exceeds threshold.
  • JIT warm-up at 09:00 ET applies to JVM-mode services only (AI Agent + warm/cold). Hot-path services ship as GraalVM native images and have no JIT to warm.
  • AI cost cap — per-tenant daily ceiling prevents runaway LLM cost.

§17

Resilience & Disaster Recovery

Resilience is layered: per-call (retry, timeout), per-dependency (circuit breaker, bulkhead), per-process (graceful shutdown, idempotent recovery), per-cluster (multi-pool isolation, anti-affinity), and per-region (cold-standby DR). The fail-closed principle (P1) trumps any retry path.

17.1 Circuit Breakers (Resilience4j)

External depCB thresholdOpen behaviour
Alpaca WS3 errors / 30 sFall back to REST polling; persists > 5 min → halt
Alpaca REST3 errors / 30 sHalt new orders; alert
Anthropic API5 errors / 60 sSentinel default ALLOW; Narrator / Reviewer skip
Auth05 errors / 60 sReject new logins; existing JWTs continue
MS SQL2 errors / 10 sFail-closed: halt all trading
Mongo5 errors / 30 sContinue (audit not critical-path); replay buffer
Kafka3 errors / 15 sFail-closed: halt

17.2 Bulkhead — Thread-Pool Isolation

  • Per-tenant pools in cold services.
  • Per-downstream pools in hot services: Alpaca-WS-bound ≠ Alpaca-REST-bound ≠ DB-bound ≠ Anthropic-bound.
  • This prevents one slow dependency (e.g., Anthropic 529) from saturating threads needed for the order path.

17.3 Idempotency

  • Alpaca orders: client_order_id = sha256(tenant_id|trade_id|action)
  • Kafka producers: enable.idempotence=true, acks=all
  • DB writes: idempotency key in metadata; UPSERT semantics
  • AI calls: deduplicated by (role, signal_id, prompt_hash) for 60 s

17.4 Retry Policy

  • Default: exp backoff 200 ms → 2 s, max 3 attempts, 100 ms jitter.
  • Order placement: NO retry on entry (signal time-sensitive); retry on exit up to 3×.
  • Audit writes: retry forever with bounded queue + DLQ.
  • AI calls: retry once on transient errors; no retry on cost-cap or content-policy errors.

17.5 DLQ + Replay

Every consumer wraps message handling in try/catch → DLQ on poison. Daily ops job inspects DLQ; replay tooling provided. DLQ messages carry failure_reason, attempt_count, original_offset.

17.6 Connection FSM Cascade

HEALTHY ─[3 consec failures]─▶ DEGRADED ─[CB open]─▶ UNHEALTHY ─[probe ok]─▶ RECOVERING ─▶ HEALTHY

Hot-path Connection FSM UNHEALTHY → Risk FSM auto-HALTS (fail-closed). Anthropic UNHEALTHY → Sentinel falls back to ALLOW (fail-open for AI). Connection FSM transitions are persisted and surfaced to the dashboard with severity.

17.7 Chaos & Soak

  • Phase 7 dedicates a build phase to chaos engineering: scripted process kill mid-trade, network partition between Decision Engine and Kafka, Anthropic 529 burst injection, MS SQL failover.
  • Per-phase soak gate (5 RTH sessions on live deps) catches issues that synthetic chaos can't reproduce.
  • 5-day continuous paper run required as Phase 8 entry gate.
